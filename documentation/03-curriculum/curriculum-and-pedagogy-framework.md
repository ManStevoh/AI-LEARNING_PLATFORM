# Curriculum And Pedagogy Framework

## Purpose

This document defines the educational foundation for the AI Learning Platform. The platform must be more than a collection of coding lessons. It should be a structured computing education pathway that grows learners from beginner digital confidence to professional software engineering and career readiness.

## Core Curriculum Thesis

Use a spiral curriculum.

Learners should revisit core computing ideas at increasing depth:

- algorithms,
- data,
- systems,
- debugging,
- collaboration,
- ethics,
- creativity,
- project development,
- AI literacy,
- security,
- software engineering,
- career readiness.

The platform should combine:

- constructionist creative projects,
- explicit instruction,
- mastery learning,
- retrieval practice,
- debugging routines,
- project-based learning,
- pair programming,
- AI-guided feedback,
- and authentic portfolio work.

## Age Bands

Age bands should guide design, but placement should depend on demonstrated mastery.

### Ages 5-7: Digital Explorer

Focus:

- device confidence,
- patterns,
- sequencing,
- directions,
- storytelling,
- unplugged logic,
- safety,
- simple block events.

Goal:

- learner believes "I can make things with technology."

### Ages 8-10: Block Creator

Focus:

- events,
- loops,
- conditionals,
- sprites,
- animations,
- simple games,
- debugging routines,
- remixing.

Goal:

- learner creates interactive artifacts with blocks.

### Ages 11-13: Block Problem Solver And Text Beginner

Focus:

- advanced blocks,
- generated JavaScript/Python,
- HTML/CSS basics,
- Python or JavaScript basics,
- functions,
- decomposition,
- simple data,
- pair programming.

Goal:

- learner understands how blocks map to text code.

### Ages 14-16: CS Foundations

Focus:

- text programming,
- algorithms,
- Boolean logic,
- databases,
- networks,
- cybersecurity,
- data representation,
- testing,
- version control basics.

Goal:

- learner can build and explain real programs.

### Ages 16-18: College And Career Bridge

Focus:

- data structures,
- APIs,
- web apps,
- databases,
- cloud basics,
- AI/data literacy,
- software engineering practices,
- capstone portfolio.

Goal:

- learner is ready for university, TVET, bootcamp, or junior technical pathway.

### Ages 18+: TVET, University, And Career

Focus:

- software development,
- ICT support,
- data analysis,
- cybersecurity,
- AI/app building,
- cloud,
- DevOps,
- entrepreneurship.

Goal:

- learner builds employable, demonstrable skills.

## Pedagogy Requirements

### Constructionism

Learners should build personally meaningful artifacts.

Requirements:

- every major unit includes a project,
- learners can remix and improve,
- projects include reflection,
- learners can share work safely where appropriate.

### PRIMM

Use Predict, Run, Investigate, Modify, Make for programming lessons.

Requirements:

- learners read code before writing full code,
- lessons include prediction tasks,
- learners inspect output,
- learners modify working examples,
- learners eventually create original artifacts.

### Use-Modify-Create

Requirements:

- beginners should start from working examples,
- gradually increase open-ended creation,
- use scaffolds that fade over time.

### Mastery Learning

Requirements:

- each lesson has objectives,
- learners complete formative checks,
- corrective practice is provided,
- mastery gates unlock next content,
- teachers can override where appropriate.

### Retrieval Practice

Requirements:

- daily review,
- previous-lesson review,
- previous-week review,
- code tracing from memory,
- vocabulary recall,
- low-stakes quizzes.

### Debugging Pedagogy

Debugging must be taught explicitly.

Routine:

1. Predict expected behavior.
2. Run the program.
3. Observe actual behavior.
4. Form a hypothesis.
5. Change one thing.
6. Test again.
7. Explain the fix.

Requirements:

- weekly bug challenges,
- AI debugging coach,
- bug pattern library,
- teacher dashboard for common bugs.

### Pair Programming

Requirements:

- driver/navigator roles,
- role rotation,
- collaboration norms,
- individual reflection,
- pair and individual assessment.

## Assessment Model

Use a balanced assessment system.

### Diagnostic Assessment

Measure:

- device fluency,
- reading level,
- math readiness,
- prior coding exposure,
- learning goals.

### Formative Assessment

Use:

- quick quizzes,
- code tracing,
- Parsons problems,
- spot-the-bug tasks,
- explain-output tasks,
- reflection prompts.

### Mastery Gates

Learners advance by competency, not just completion.

### Project Rubrics

Assess:

- functionality,
- computational concepts,
- design process,
- originality,
- testing,
- debugging,
- communication,
- reflection.

### Portfolio Evidence

Every major level should produce artifacts:

- game,
- animation,
- website,
- data project,
- API app,
- team project,
- capstone.

### Career Assessment

Assess:

- GitHub readiness,
- README quality,
- tests,
- deployed project,
- code review,
- interview reasoning,
- communication.

## AI Teaching Requirements

AI should:

- provide hints before answers,
- adapt to learner level,
- reference lesson context,
- identify misconceptions,
- encourage debugging,
- suggest retrieval practice,
- help teachers intervene,
- generate practice variations,
- support project reflection.

AI should not:

- complete graded work without scaffolding,
- bypass learning objectives,
- hallucinate curriculum claims,
- provide unsafe or age-inappropriate content.

- worked example,

- debugging challenge,
- project,
- formative quiz,
- mastery gate,
- reflection,
- teacher notes,
- AI prompt behaviors,
- accessibility considerations,
- offline option where possible.
