# Assessment Framework And Rubrics

## Purpose

This document defines how the platform assesses learning from Level 1 block coding to Level 5 professional projects.

The platform should assess demonstrated competence, not only lesson completion.

## Assessment Principles

- Assessment should support learning, not only grading.
- Mastery requires evidence across time and contexts.
- AI may assist assessment but high-stakes outcomes require human oversight.
- Rubrics should be clear before work starts.
- Feedback should be actionable and age-appropriate.
- Assessment should map to competencies and the Learning Genome.

## Assessment Types

### Diagnostic

Purpose:

- understand starting level,
- recommend path,
- identify prerequisites.

Examples:

- placement quiz,
- simple coding task,
- block logic challenge.

### Formative

Purpose:

- guide learning during lessons.

Examples:

- quick checks,
- hints,
- practice tasks,
- debugging questions,
- AI explanations.

### Summative

Purpose:

- evaluate a unit, level, or certificate.

Examples:

- final project,
- practical exam,
- portfolio review,
- coding assessment.

### Authentic Assessment

Purpose:

- evaluate real-world ability.

Examples:

- build a working app,
- design an API,
- debug a broken project,
- explain architecture decisions.

## Evidence Strength

Weak evidence:

- watching a video,
- opening a lesson,
- reading notes.

Medium evidence:

- passing a quiz,
- completing guided practice,
- fixing a known bug.

Strong evidence:

- completing a project,
- solving a transfer task,
- explaining reasoning,
- passing teacher-reviewed rubric,
- applying skill in a new context.

## Rubric Dimensions

Common dimensions:

- correctness,
- completeness,
- creativity,
- code/logic clarity,
- debugging,
- use of target concepts,
- security/safety where relevant,
- testing,
- documentation,
- reflection.

## Level 1 Block Coding Rubric

Dimensions:

- uses required blocks,
- project runs,
- event logic works,
- sequence is clear,
- loop/condition use matches objective,
- sprite/stage behavior is understandable,
- learner can explain project,
- creativity/theme.

Levels:

- Beginning,
- Developing,
- Proficient,
- Advanced.

Example:

```text
Loop Use
Beginning: loop missing or disconnected.
Developing: loop exists but does not fully support the goal.
Proficient: loop works correctly for repeated action.
Advanced: loop is used creatively with variables/events.
```

## Coding Project Rubric

Dimensions:

- functional correctness,
- requirements coverage,
- code readability,
- structure,
- error handling,
- validation/security,
- tests,
- documentation,
- deployment readiness.

## AI Assessment Rules

AI can:

- draft feedback,
- identify likely misconceptions,
- map submissions to competencies,
- suggest rubric scores for review,
- generate practice based on gaps.

AI must not:

- make final high-stakes certification decisions alone,
- leak hidden tests,
- provide full solutions during active assessments,
- fabricate evidence.

## Assessment Integrity

Controls:

- assessment mode,
- limited AI help,
- hidden tests,
- randomized variations,
- submission snapshots,
- audit logs,
- teacher review,
- plagiarism/originality checks later.

## Feedback Requirements

Feedback should include:

- what worked,
- what needs improvement,
- one or two next actions,
- relevant lesson links,
- encouragement without false praise,
- competency mapping where useful.

## Certificate Requirements

Certificates should require:

- required competencies,
- minimum evidence count,
- project artifact,
- assessment result,
- teacher or approved review where high-stakes.

## Acceptance Criteria

- every assessment maps to competencies,
- every project has a rubric,
- AI feedback is labeled,
- high-stakes outcomes require human oversight,
- learners can understand next steps,
- teachers can override AI scoring.
